## Project Overview

World Monitor is a real-time OSINT (Open Source Intelligence) dashboard built with **vanilla TypeScript** (no UI framework), **MapLibre GL + deck.gl** for 3D map rendering, and a custom Proto-first HTTP RPC framework called **Sebuf** for all API communication. It runs as a web app (Vite), desktop app (Tauri v2), and PWA.

## Common Commands

```bash
# Install everything (buf CLI, sebuf plugins, npm deps, Playwright browsers)
make install

# Dev server (http://localhost:3000)
npm run dev              # full (geopolitics) variant
npm run dev:tech         # tech variant
npm run dev:finance      # finance variant

# Build
npm run build            # full variant
npm run build:tech
npm run build:finance

# Type checking
npm run typecheck        # frontend (src/)
npm run typecheck:api    # API layer (api/ + server/ + src/generated/)
npm run typecheck:all    # both

# Tests
npm run test:data        # data integrity tests (tsx --test)
npm run test:sidecar     # sidecar + API unit tests (node --test)
npm run test:e2e         # all Playwright e2e tests
npm run test:e2e:full    # e2e for full variant only

# Proto workflow
make generate            # regenerate TS clients/servers + OpenAPI from protos
make lint                # lint proto files
make check               # lint + generate
make breaking            # check for breaking proto changes vs main
make format              # format proto files

# Desktop
npm run desktop:dev      # Tauri dev mode with devtools
```

## Pre-push Hook

The `.husky/pre-push` hook runs `typecheck`, `typecheck:api`, and `version:check` before every push.

## Architecture

### Variant System

One codebase produces four app variants via `VITE_VARIANT` env var: `full` (default/geopolitics), `tech`, `finance`, `happy`. Variants share all code but differ in default panels, map layers, and RSS feeds. Variant resolution: `src/config/variant.ts`. Panel/layer configs per variant: `src/config/panels.ts`.

### Sebuf RPC Framework (Critical Path)

All JSON API endpoints use Sebuf. The flow:

1. **Proto definitions** in `proto/worldmonitor/{domain}/v1/` define services + messages
2. **`make generate`** produces typed TS clients (`src/generated/client/`) and server stubs (`src/generated/server/`) plus OpenAPI docs (`docs/api/`)
3. **Handlers** in `server/worldmonitor/{domain}/v1/handler.ts` implement the generated interface
4. **Edge gateway** at `api/[domain]/v1/[rpc].ts` registers all 21 domain handlers and routes requests
5. **Frontend service wrappers** in `src/services/{domain}/` wrap the generated client for app use

There are 21 domain services: aviation, climate, conflict, cyber, displacement, economic, giving, infrastructure, intelligence, maritime, market, military, news, positive-events, prediction, research, seismology, supply-chain, trade, unrest, wildfire.

**Never edit `src/generated/`** — these files are auto-generated by `make generate`.

### Adding a New RPC Method

1. Add/edit `.proto` files in `proto/worldmonitor/{domain}/v1/`
2. Run `make generate`
3. Implement the handler in `server/worldmonitor/{domain}/v1/` (TypeScript will error until you do — the generated interface enforces the contract)
4. The client stub is auto-generated; wrap it in `src/services/{domain}/`
5. For a new service, also register it in `api/[domain]/v1/[rpc].ts` and `vite.config.ts`

### Proto Conventions

- Time fields: `int64` (Unix ms), never `google.protobuf.Timestamp`. Always annotate with `[(sebuf.http.int64_encoding) = INT64_ENCODING_NUMBER]`
- Every RPC: `option (sebuf.http.config) = {path: "/verb-noun"}`
- Reuse `core/v1` types: `GeoCoordinates`, `BoundingBox`, `TimeRange`, `PaginationRequest/Response`
- `buf lint` requires comments on all messages, fields, services, RPCs, and enum values
- One file per message type, one file per RPC pair, `service.proto` for the service definition

### Frontend Architecture

- **No framework** — vanilla TypeScript with manual DOM manipulation
- `src/App.ts` — main application class, orchestrates panels, map, data loading
- `src/main.ts` — entry point (Sentry init, app bootstrap)
- `src/components/Panel.ts` — base class for all ~50 dashboard panels
- `src/components/Map.ts` + `DeckGLMap.ts` + `MapContainer.ts` — map rendering stack
- `src/services/` — data fetching modules (~100 files), each wrapping a sebuf client or external API
- `src/config/` — static data (feeds, geo coords, military bases, pipelines, ports, variant configs)
- `src/locales/` — i18n JSON files (16 languages), lazy-loaded
- `src/workers/` — Web Workers for ML analysis
- `src/app/` — App sub-managers (panel-layout, data-loader, event-handlers, search, refresh-scheduler)
- Path alias: `@/*` maps to `src/*`

### Backend / API Layer

- `api/` — Vercel Edge Functions: the sebuf gateway (`[domain]/v1/[rpc].ts`) plus legacy endpoints (RSS proxy, YouTube embed, Telegram feed, etc.)
- `server/` — sebuf handler implementations + shared utilities (`_shared/` for rate limiting, response headers, caching)
- Legacy `api/*.js` endpoints exist for non-JSON payloads (XML feeds, binary data, HTML embeds) but all new JSON endpoints must use Sebuf

### Desktop App (Tauri v2)

- `src-tauri/` — Rust app + Node.js sidecar that runs all API handlers locally
- Settings stored in OS keychain (consolidated vault pattern)
- Cloud fallback when local handlers fail

### TypeScript Config

- `tsconfig.json` — frontend (`src/`), strict mode, `noUnusedLocals`, `noUnusedParameters`, `noUncheckedIndexedAccess`
- `tsconfig.api.json` — extends base, covers `api/`, `server/`, `src/generated/`

### Environment

- All API keys optional — app runs without them, features degrade gracefully
- Copy `.env.example` to `.env.local` for local development
- `VITE_`-prefixed vars are client-side; others are server-side only
